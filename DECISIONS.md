# Architecture Decisions Log

Формат: проблема → рассмотренные варианты → решение → почему.

## ADR-001: Разделение SQLAlchemy моделей и Pydantic-схем

**Проблема:** нужно ли использовать одну и ту же сущность для описания структуры таблицы в БД и для контракта API (вход/выход запросов)?

**Решение:** модели (`models/`) и схемы (`schemas/`) — два разных слоя, не смешиваются.

**Почему:**
- Модель описывает структуру таблицы, схема — публичный контракт API. Это два разных повода для изменения (SRP).
- Модель может содержать поля, которые не должны уходить наружу.
- Правила валидации при создании/выводе объекта могут отличаться от ограничений в БД.
- Изменение структуры таблицы не должно незаметно менять контракт API.

## ADR-002: Repository Pattern поверх прямых запросов в роутере

**Проблема:** если писать SQLAlchemy-запросы прямо в роутерах, логика доступа к БД дублируется между эндпоинтами и её невозможно тестировать изолированно от HTTP-слоя.

**Решение:** вся работа с БД инкапсулируется в классах-репозиториях (`repositories/`).

**Почему:**
- Централизация БД-логики в одном месте (DRY).
- Роутер/сервис получает простой интерфейс ("получить", "добавить", "удалить"), не зная деталей реализации.
- Репозиторий можно тестировать отдельно, подменять фейковой реализацией в unit-тестах.

## ADR-003: Repository работает только с ORM-моделями, не с Pydantic-схемами

**Проблема:** должен ли Repository принимать/возвращать Pydantic-схемы напрямую?

**Рассмотренные варианты:**
- Repository принимает Pydantic-схему (например, `CategoryCreate`) как параметр.
- Repository принимает только примитивные параметры (`name: str`) и сам создаёт ORM-объект внутри.

**Решение:** второй вариант — только примитивные параметры на входе, только ORM-модели на выходе.

**Почему:**
- Repository — нижний слой, ближе к БД. Если он будет зависеть от Pydantic-схем (верхний слой), зависимости пойдут "вверх", что нарушает чистоту архитектуры и мешает переиспользовать репозиторий вне FastAPI (например, в скриптах, фоновых задачах, тестах).
- База данных ничего не знает про HTTP/Pydantic — Repository должен отражать это.

## ADR-004: AsyncSession передаётся через конструктор, без `Depends` внутри Repository

**Проблема:** `Depends(get_db)` — механизм FastAPI Dependency Injection, работающий только внутри контекста обработки HTTP-запроса.

**Решение:** `CategoryRepository.__init__(self, session: AsyncSession)` — простой параметр, без `Depends`.

**Почему:**
- Жёсткая привязка к `Depends` делает репозиторий неотделимым от FastAPI — его нельзя создать в unit-тесте или фоновом скрипте без поднятия всего приложения.
- `Depends(get_db)` уместен на уровне роутера/сервиса, откуда уже готовая сессия передаётся в конструктор репозитория.

## ADR-005: Возвращаемый тип `get_categories` — `Sequence`, а не `List`

**Проблема:** метод `.scalars().all()` в SQLAlchemy 2.0 объявлен с типом `Sequence[T]`, а не `List[T]`.

**Решение:** использовать `Sequence[CategoryModel]` в сигнатуре, полагаясь на тип, который реально гарантирует библиотека.

**Почему:**
- `Sequence` — более общий/абстрактный тип, честно отражающий контракт SQLAlchemy.
- Вызывающему коду в этом проекте достаточно возможностей `Sequence` (итерация, индексация, `len()`) — сужать тип до `List` искусственно не нужно (YAGNI).
- Более гибко к изменениям в библиотеке в будущем.

## ADR-006: `delete_category` возвращает удалённый объект, а не `bool`

**Проблема:** что должен возвращать метод удаления — факт успеха (`bool`) или сам удалённый объект?

**Решение:** `Optional[CategoryModel]` — сам удалённый объект, либо `None`, если объект не найден.

**Почему:**
- Если вернуть только `bool`, вызывающему коду (например, для логирования) придётся делать дополнительный запрос, чтобы узнать, что именно было удалено.
- Возврат самого объекта даёт вызывающему коду всю нужную информацию без лишнего похода в БД.
- `Optional` явно отражает случай "объект с таким id не найден" (пока без выбрасывания кастомных исключений — это будет рассмотрено отдельно на этапе обработки ошибок).

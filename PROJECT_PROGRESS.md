# Архитектурные решения

Мы придерживаемся следующих правил:

- Repository Pattern
- Service Layer
- SQL из роутеров не пишем
- Изменения структуры БД только через Alembic
- SQLAlchemy модели и Pydantic схемы разделены
- Не создавать лишние абстракции заранее (KISS)
- Не писать функциональность "на будущее" (YAGNI)
- Избегать дублирования (DRY)

# Логирование

Используем Loguru.

На текущем этапе сознательно ещё не добавляли логи, так как отсутствует бизнес-логика.

При реализации Repository и CRUD необходимо обсудить:

- какие действия логировать;
- на каком уровне логировать;
- какие сообщения должны попадать в логи.

# Формат обучения

ChatGPT/Claude выступает как Senior Backend Developer и наставник.

Правила обучения:

1. Сначала обсуждаем проблему.
2. Затем проектируем решение.
3. Потом я предлагаю своё решение.
4. После обсуждения пишем код.
5. Для небольших задач сначала код пишу я.
6. После каждого этапа проводится code review.
7. После крупных этапов проводится мини-собеседование.
8. Если вижу архитектурную проблему — сначала обсуждаем её, а потом меняем проект.
9. Цель — понимать причины решений, а не копировать код.

# Текущий этап

## Repository Pattern

Мы ещё не написали ни одной строки кода.

На данный момент обсуждено:

- зачем нужен Repository;
- почему работа с БД не должна находиться в роутерах;
- почему Repository уменьшает дублирование кода;
- почему Repository отвечает только за работу с БД;
- почему AsyncSession будет передаваться через конструктор Repository.

Следующий шаг:
